Single Prisma project rooted at `prisma/schema.prisma` targeting a PostgreSQL database via `DATABASE_URL`/`DIRECT_URL` env vars, with binary targets `native` and `linux-musl-openssl-3.0.x`. The model is split across ~19 domain-scoped files under `schema/` (`system`, `user`, `opmc`, `contractor`, `project-core`, `project-finance`, `project-workflow`, `inventory`, `stock-management`, `service-order`, `helpdesk`, `gis`, `survey`, `vehicle-management`, `petty-cash`, `nexus`, `agent-sync`, plus shared `enums.prisma` and `_base.prisma`) — each file declares its own models and relations, then the root `schema.prisma` re-exports them so Prisma generates one unified client.

Evolution is tracked in timestamped directories under `migrations/` (e.g. `20260109164529_add_three_tier_approval_system`), generated by `prisma migrate dev`; `migration_lock.toml` pins the provider to PostgreSQL. Seed data lives alongside the schema: Node-based seeds (`seed.js`, `seed-authorities.ts`, `seed-helpdesk.js`, `seed-workflows.ts`) use the generated Prisma Client to upsert reference data (super admin, OPMCs, stores, authorities, workflows), while raw SQL fixtures (`insert_test_users.sql`, `seed-sod-revenue.sql`, `data/boc_branches.js`) supplement bulk loads. A helper `generate_hash.js` produces bcrypt hashes consumed by seeds.
