Flat Next.js `public/` directory with no build step — every file is served verbatim by the host. Two parallel Chrome-extension distributions coexist: `extension/` (source) and `slt-bridge/` (development build that additionally grants `cookies` permission and whitelists `https://sltserp.vercel.app/*`). Each extension follows the same MV3 layout — `manifest.json`, a `background.service_worker`, a `popup.html`+`popup.js` action UI, three content scripts (`content-slt.js` for slt.lk pages, `content-erp.js` injected at `document_start`, `content-main.js` in the MAIN world), plus an icon. Versioned prebuilt extensions are shipped as zip archives under `extension-builds/` (phoenix-*, phoenix-elite-*, phoenix-omni-*, slt-bridge-*). The root also holds the signed `.crx` / `.zip` / `.pem` bundle of the latest bridge, a PWA `sw.js`, Excel `.xlsm` invoice templates under `templates/`, a Windows installer under `downloads/`, and generic SVG/PNG logos. There is no internal code logic — this module is purely a content-addressable asset store consumed by the frontend app and end users.
