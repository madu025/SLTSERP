This monorepo uses the standard npm ecosystem for JavaScript/TypeScript dependency management. There is no Go, Python, or other language-specific lockfile in use at the repository root — all application code is Node.js (Next.js 16) and dependencies are declared exclusively via npm.

**System used**
- Package manager: npm (lockfileVersion 3)
- Lockfile: `package-lock.json` committed to version control alongside `package.json`
- No vendoring of node_modules; packages are installed from the public npm registry (`https://registry.npmjs.org`) as evidenced by every entry in `package-lock.json` resolving to a `.tgz` under that URL.
- No private npm registry, `.npmrc`, `GOPRIVATE`, or vendor directory was found anywhere in the tree.

**Key files**
- `package.json` — declares runtime and dev dependencies with caret ranges (e.g. `^5.x`, `^6.x`), pins Next.js, React, Prisma, BullMQ, ioredis, Radix UI, OpenLayers, etc. Also defines `engines.node = "24.x"` and a `postinstall` hook that runs `prisma generate`.
- `package-lock.json` — fully committed deterministic lockfile pinning every transitive dependency to an exact tarball SHA, ensuring reproducible installs across CI and production.
- `Dockerfile` / `docker-compose*.yml` — pull images from Docker Hub (no custom registry); container builds rely on the same npm install flow.

**Architecture & conventions**
- Single-root dependency graph: there is only one `package.json`; subdirectories do not maintain independent npm workspaces or pnpm/yarn projects.
- Versioning strategy: caret (`^`) ranges are used throughout, allowing minor/patch upgrades while keeping major versions locked. This balances stability with the ability to receive security patches automatically.
- Build-time integration: `next build` depends on `npx prisma generate` running first (via both the `build` script and `postinstall`), so the Prisma client must be regenerated whenever `prisma/schema` changes.
- Dev-only tooling (Playwright, ESLint, TypeScript, tsx) is separated into `devDependencies` and does not ship to production.

**Rules developers should follow**
1. Always run `npm install` (or `npm ci` in CI) after pulling changes — never delete `node_modules` without re-running install, because the lockfile drives determinism.
2. When adding a new dependency, declare it in the correct section (`dependencies` vs `devDependencies`) and commit the updated `package-lock.json`.
3. Prefer caret ranges (`^x.y.z`) over tilde or exact pins unless a specific bug fix requires it; coordinate major-version bumps together since many packages (Next, React, Prisma) are tightly coupled.
4. Do not add a local `.npmrc` pointing to a private registry without updating this card — no such configuration currently exists.
5. After changing any file under `prisma/schema`, run `npm run build` (which triggers `prisma generate`) before committing, so the generated client stays in sync.
